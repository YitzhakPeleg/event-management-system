Project Architecture

model/

Plain Java objects (POJOs): User, Hall, Event, Participant, Registration

db/ (DAO layer)

One class per table. All SQL queries. Prepared statements (no SQL injection).

service/

Business logic: capacity check, validation before DB, QR token generation

api/

JSON Servlets called by Android. One servlet per resource (events, halls...)

web/ + JSP

HTML interface for browser users. Calls the same service layer.

util/

EncryptionUtil (AES), HashUtil (SHA-256), ValidationUtil, DBConnection

Security Features

🔐 Password Hashing

Passwords hashed with SHA-256 before storage.
Never stored as plain text.

🔒 AES Encryption

Participant email and phone encrypted before DB.
Decrypted only when needed.

🛡️ SQL Injection Prevention

All queries use PreparedStatement.
User input never concatenated into SQL.

✅ Input Validation

ValidationUtil checks email format, phone, date, capacity before every action.

Bonus Features

📷 Camera — QR Scanning

Admin uses the physical camera on the Android device to scan participant QR codes at the entrance. Uses the ZXing barcode library. This is a hardware component (סעיף 10).

🔑 QR Code Generation

On every registration, a unique UUID-based token is generated and stored. The Android app renders it as a QR code bitmap using ZXing. The token is validated server-side on scan.

🔒 AES + SHA-256

Double-layer security: AES encryption for personal data fields, SHA-256 hashing for passwords. This covers the encryption bonus (סעיף 9).
